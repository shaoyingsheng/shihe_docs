代码规范

变量名命名

变量名尽量见名识义, 首字母小写，后续驼峰格式: 
userName (用户名称)
idList (id列表)

代码文件命名

也要见名识义，类似于变量名命名，但首字母大写:
TimeUtil.py (时间处理代码)
DatabaseApi.py (数据库接口代码)
UserPage.py (用户界面代码)

代码编写规范

代码清晰整洁，不无缘无故空行;
多注释，一般每几行就加一行注释(最小块的功能);
代码模块写在对应的代码文件里:
比如UserPage.py里需要用到getCurTimeStamp()函数(获取当前时间戳),
这个getCurTimeStamp的定义代码不应该写在UserPage.py里，而应该写
在TimeUtil.py里，然后在UserPage.py里import再调用

代码文件分类

代码文件要分类，保存在对应类的目录下，做到整齐清晰:
TimeUtil.js 属于工具类，要放在util文件夹下(文件夹命名全部小写)
DatabaseApi.js 属于接口，要放在api文件夹下

Git操作

平时开发时，在自己的分支上开发，一开始没有分支，所以使用

git checkout -b sys

新建一个自己的分支"sys"。可以使用

git branch

查看当前所处的分支，可以看到处于分支"sys"下。一顿编辑结束后使用

git add .

把修改提交到暂存区，再使用

git commit -m "本次修改说明"

提交到本地仓库，此时准备把新版本提交到远端"sys"仓库

git push

但历史第一次提交会失败，因为远端没有"sys"仓库，git会给一个提示:

git push -upstream ... ...

复制它的指令再次提交就行了。此后在"sys"分支提交直接git push就好了。

此时我们每次只更新了自己的"sys"仓库，什么时候更新master分支呢？

如果确认当前模块功能已经完成，且把自己的代码和master代码合并之后不会报错，就可以更新master分支，也就是说，master分支必须保证是可运行状态，没有报错。

怎么把自己当前分支的代码合并到master分支呢？

首先，在自己的分支"sys"下使用指令：

git rebase master

该指令把master分支的最新代码合并到当前分支。这个合并可能导致冲突，比如A之前改了公共模块T.js(包含时间处理函数，大家通用的代码块),我不知道A改了，所以在自己分支下也改了T.js，这时候可能导致冲突(因此，最好在每次进入自己分支准备写代码前，先git rebase master，把别人在master上更新的公共模块更新到自己当前分支。此外，更新过公共模块的人，最好在群里说一下，让大家先git rebase master一下)。如果有冲突，无关紧要的冲突直接改，有问题的冲突商量着解决。

合并完代码，解决完冲突(如果有)，确认全部代码不会报错后，切换到master分支：

git checkout master

把自己分支的代码更新到master:

git merge sys

最后提交到远端的master分支:

git push
